# tech-poc-planner 出力フォーマット

## フロントマター（YAMLメタ情報 + 検証項目一覧 + 懸念事項）

```yaml
---
title: "{想定タイトル}"
date: {YYYY-MM-DD}
author: tech-poc-planner
tags: [{関連タグをカンマ区切り}]
target_audience: "{想定読者}"
status: draft
material_source: "{素材シートのパス}"
theme: "{テーマ名}"
poc_directory: "~/works/poc-something/{YYYY-MM-DD}_{テーマのkebab-case}/"
feasibility_assessment:
  freshness_pressure: high|medium|low
  verification_cost: low|medium|high
  blocker_severity: none|workaround|critical
  article_value: high|medium|low
  material_readiness: sufficient|partial|insufficient
  decision: go|conditional_go|no_go
  decision_reason: "{判定理由を1文で}"
  verification_strategy: minimal|standard|full
estimated_verification_hours: {合計見積もり時間（サマリー作成含む全項目の総計）}
verification_items:
  - id: 1
    title: "{検証項目タイトル}"
    priority: required|recommended
    estimated_minutes: {分}
    status: pending
    depends_on: []
  - id: 2
    title: "{検証項目タイトル}"
    priority: required|recommended
    estimated_minutes: {分}
    status: pending
    depends_on: [1]
    skip_if: "{推奨項目のみ: スキップ条件}"
  - ...
concerns:
  - type: paid_api|account_required|special_hardware|license
    service: "{サービス名}"
    detail: "{詳細}"
    estimated_cost: "{概算コスト or 無料枠あり}"
    workaround: "{回避策（モック利用、無料枠、代替サービス等）}"
  - ...
---
```

※ `title`, `tags`, `target_audience` は素材シートの情報から設定する。検証完了後に最終調整する。
※ `poc_directory` はtech-poc-runnerが検証作業を行うディレクトリ。テーマに応じて適切なパスを設定する。
※ `verification_items` の各項目には連番 `id` を付与し、本文の検証詳細セクションと対応させる。

## 出力ファイルの全体構造

```markdown
---
{YAMLフロントマター: メタ情報 + 検証項目一覧 + 懸念事項}
---

# {タイトル}

## はじめに
{記事本文...}

## {技術解説セクション}
{記事本文...}

## セットアップ
{記事本文...}

## 基本操作
{記事本文...}

## 実践: {ユースケース}
{記事本文...}

## まとめ
{記事本文...}

## 参考資料
{参考URL一覧}

---

## 技術検証計画（詳細）

### 概要
{検証の目的・全体方針を1〜3文で}

### 環境構成
{テーブル形式で必要な環境を記載}

### 素材シートとの乖離（該当する場合のみ）
{乖離テーブル}

### 項目1: {タイトル}（{必須|推奨}・{見積もり時間}）
{目的・手順・コード例・確認ポイント・参考URL}

### 項目2: ...
...

### 所要時間見積もり
{フェーズ別テーブル}

### 前提条件
{APIキー、Docker等の前提}
```

この構造により:
- **記事草案**（上半分）: 読者向けの記事ドラフトとして独立して読める
- **検証計画**（下半分）: tech-poc-runnerが実行手順書として使える
- 両者が1ファイルに統合されているため、検証結果を記事のTBDに反映しやすい

## 検証項目の詳細展開テンプレート

### 依存関係図

```
項目1（環境構築）
  ├→ 項目2（基本動作）
  │    ├→ 項目3（応用機能A）
  │    └→ 項目4（応用機能B）
  └→ 項目5（パフォーマンス）← 項目2完了後
項目6（ドキュメント確認）← 独立
```

**クリティカルパス**: 項目1 → 項目2 → 項目3（最長経路: {N}分）

依存関係のルール:
- **ゲート項目**（他の全項目が依存する項目）を明示する。ゲート項目が失敗した場合、後続項目は全てスキップ
- 独立して実行可能な項目は「独立」と明記する（並列実行可能）
- フロントマターの `verification_items` にも `depends_on` フィールドを追加する

### スキップ条件（推奨項目のみ）

推奨（recommended）項目には、**スキップしてよい条件**を明記する:

```markdown
**スキップ条件**: {この条件に該当する場合はスキップ可}
**スキップ時の影響**: {記事のどの部分が書けなくなるか}
**代替手段**: {スキップした場合の記事上の対処法 — 「TBDのまま公開」「セクション削除」「公式ドキュメント引用で代替」等}
```

### 各検証項目に必須の記載事項

```markdown
### 項目N: {タイトル}（{必須|推奨}・{見積もり時間}）

**目的**: {この検証で何を確認するか — 1文}

**手順**:
1. {具体的なコマンドまたはコード}
2. ...

**コード例**:
```{language}
// 公式ドキュメントで確認済みのAPI
{実行可能なコードスニペット}
```

**確認ポイント**:
- {何を見て成功/失敗を判断するか}
- {期待される出力・戻り値}

**参考**: {公式ドキュメントURL}
```

### 素材シートとの乖離テーブル（該当する場合）

| 素材シートの記載 | 公式API（正） | 備考 |
|-----------------|--------------|------|
| `旧API()` | `正しいAPI()` | 差異の説明 |
```

```markdown

| フェーズ | 時間 |

```

## TBDの記載ルール

技術検証が必要な箇所には以下の形式でTBDマーカーを挿入する:

```markdown
<!-- TBD[id:N]: {検証が必要な内容の説明} -->
<!-- 検証方法: {どうやって確認するか} -->
```

`id:N` はフロントマターの `verification_items` の `id` と対応させる。1つのTBDが複数の検証項目に関連する場合はカンマ区切り（`id:2,3`）。検証項目に直接対応しないTBD（公式ドキュメント確認のみ等）は `id:0` とする。

### TBDにすべき箇所の判定基準

以下に該当する箇所はTBDとする:
- 素材シートで「動作確認状況: 未確認」のコードサンプル内の詳細挙動
- 素材シートの「要追加確認事項」に関連する記述
- APIのメソッドシグネチャ・引数・戻り値で公式ドキュメントの裏取りが不十分なもの
- エラーハンドリング・エッジケースの挙動
- パフォーマンス特性・制限事項

### TBDにしない箇所（そのまま書く）

以下は素材シートの情報で十分書ける:
- 記事の導入・背景説明（裏どり済み事実を使用）
- 技術の概要・アーキテクチャ説明
- 比較表・文脈情報
- コードサンプルの大枠（細部の型情報等はTBD）
- まとめ・結論
