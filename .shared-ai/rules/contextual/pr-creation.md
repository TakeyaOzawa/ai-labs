# PR作成ガイド

## 基本方針

KiroでPRを作成する際は、以下の手順と基準に従ってください。

## PR作成手順

### 1. 事前準備

- 作業内容が明確に定義されていることを確認
- 関連するテストが実装されていることを確認
- コードの品質チェックが完了していることを確認（リンター、静的解析）

### 2. ブランチ作成

- タスクIDがある場合はそれを含むブランチ名を使用
- 例: `feature/{TASK-ID}`, `feature/{TASK-ID}-{description}`
- タスクIDが不明な場合は機能名を含む分かりやすい名前を使用

### 3. コミットメッセージ

- 日本語で記載
- Conventional Commits準拠のプレフィックスを使用
- 変更内容を簡潔に説明
- 例: `feat: キュー管理機能を追加`, `fix: バリデーションエラーを修正`

#### spec_typeとの対応

コミットメッセージのプレフィックスは、specの `spec_type` と一致させる:

| spec_type | コミットプレフィックス | 用途                   |
| --------- | ---------------------- | ---------------------- |
| feat      | `feat:`                | 新規機能追加; 機能変更 |
| fix       | `fix:`                 | 不具合修正             |
| refactor  | `refactor:`            | コード構造改善         |
| perf      | `perf:`                | パフォーマンス改善     |
| docs      | `docs:`                | ドキュメント整備       |

### 4. PR作成時の必須項目

#### specのstatus更新（specを使用している場合）

PR作成前に、対応するspecの `requirements.md` のfront-matterで `status` を `frozen` に更新する:

```yaml
status: frozen # active → frozen に変更
updated_at: { 今日の日付 }
updated_by: { 更新者 }
```

同様に `design.md`, `tasks.md` のfront-matterも `frozen` に更新する。

#### タイトル

- プレフィックスを使用: `feat:`, `fix:`, `refactor:`, `test:`, `docs:`
- 変更内容を簡潔に表現

#### 説明文

プロジェクトにPRテンプレート（`.github/pull_request_template.md`）がある場合はそれに従う。
テンプレートがない場合は以下の構成で記載:

```markdown
## 概要

- このプルリクで何をしたのかを具体的に記載

## やらないこと

- このプルリクでやらないことは何か？（あれば）

## 動作確認

- どのような動作確認を行ったのか？結果はどうか？

## その他

- レビュワーへの参考情報（実装上の懸念点や注意点などあれば記載）
```

## 品質基準

### テスト

- 新機能には対応するテストを必須で実装
- 既存テストが破綻していないことを確認
- カバレッジの維持・向上

## 自動化ツール活用

- CIによる自動テスト実行を活用
- コードフォーマッターによる自動フォーマット
- 静的解析ツールによる品質チェック
- 静的解析ツールの結果を確認してからPR作成

## プロジェクト固有のカスタマイズ

ブランチ命名規則、タスクID体系、外部ツール連携（Notion等）はプロジェクトの `.kiro/steering/pr-creation-guide.md` で追加定義すること。
