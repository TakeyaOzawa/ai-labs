# README 記述ガイド

プロジェクトやリポジトリのREADME.mdを作成する際は、本ガイドの記述ルールに従ってください。
本フォーマットはIEEE 1063:2001（Software User Documentation）の構造を基盤に、ISO/IEC 26514:2022のタスク指向設計とITIL v4の構成管理観点を組み合わせたものです。

## 準拠規格

- IEEE 1063:2001 — Standard for Software User Documentation（ユーザードキュメントの構成）
- ISO/IEC 26514:2022 — Design and development of information for users（タスク指向の情報設計）
- ITIL v4 — Service Design / Configuration Management（構成アイテム管理、依存関係の可視化）
- 簡略化の方針: IEEE 1063のセクション構成を、開発者向けREADMEに適した粒度に変換。

## 設計思想

### IEEE 1063 からの採用要素

| IEEE 1063 セクション | README での対応 | 採用理由 |
|---|---|---|
| Identification | §1 概要 | ドキュメントの目的・スコープを明確化 |
| Prerequisites | §2 前提条件 | 利用開始前に必要な環境を明示 |
| Installation | §3 セットアップ | タスク指向のインストール手順 |
| Orientation | §4 アーキテクチャ | システム全体像の把握を支援 |
| Reference | §5 ディレクトリ構成 | 構成要素の一覧と関係性 |
| Procedures | §6 運用手順 | タスク指向の操作ガイド |
| Troubleshooting | §7 トラブルシューティング | 問題解決の自己支援 |
| Appendix | §8〜 | 制約事項、関連ドキュメント、変更履歴 |

### ISO/IEC 26514 からの採用要素

- **タスク指向**: 「何ができるか」ではなく「何をしたいか」を起点に構成する
- **対象読者の明示**: 読者のスキルレベル・目的に応じた情報の粒度を設定する
- **段階的開示**: 概要→詳細の順で情報を提示し、必要な深さまで掘り下げられる構造にする

### ITIL v4 からの採用要素

- **構成アイテム（CI）一覧**: ディレクトリ構成をCI一覧として整理し、各要素の責務を明記する
- **依存関係の可視化**: コンポーネント間の参照関係を図示する
- **変更管理**: 変更履歴セクションで変更の追跡性を確保する

## 記述ルール

### 構成に関するルール

1. **セクション番号は1始まりの連番**とする。不要なセクションは省略してよいが、番号は詰める
2. **§1〜§3は必須**。§4以降はプロジェクト規模に応じて省略可
3. **小規模プロジェクト**（ファイル数10未満）の場合、§4 アーキテクチャと§5 ディレクトリ構成は統合してよい
4. **§8 既知の制約**は該当がなければセクションごと省略する

### 記載スタイル

1. **タスク指向で書く**: 「このモジュールは〜する」ではなく「〜したい場合は〜する」
2. **テーブルを活用する**: 比較・一覧はテーブルで整理し、散文を避ける
3. **コマンドブロックを含める**: 手順にはコピー可能なコマンドを必ず付記する
4. **段階的開示**: 概要→詳細の順。詳細は別ドキュメントへのリンクで対応してよい
5. **冗長な説明を避ける**: 1セクション内で同じ情報を繰り返さない。他セクションへの参照で代替する

### 対象読者に応じた調整

| 読者 | 重点セクション | 省略可能 |
|---|---|---|
| 初回利用者 | §2, §3 | §4, §5 |
| 日常運用者 | §6, §7 | §3 |
| 開発者 | §4, §5, §6 | — |
| 障害対応者 | §7, §8 | §3, §4 |

### 他のフォーマットガイドとの関係

| ガイド | 用途 | 本ガイドとの関係 |

### プロジェクトルートREADME vs サブディレクトリREADME

READMEは階層的に配置し、段階的開示を実現する。

| 配置 | 責務 | 対象読者 | 必須セクション |
|---|---|---|---|
| プロジェクトルート `~/README.md` | 全体のアーキテクチャ・セットアップ・運用 | 環境を初めて使う人、全体像を把握したい人 | §1〜§3（概要・前提・セットアップ）+ §4（アーキテクチャ） |
| サブディレクトリ `{dir}/README.md` | そのディレクトリ内部の詳細な配置基準・更新手順 | ファイルを追加・編集する人 | §1（概要）+ 配置判断基準 + 更新手順 |

#### プロジェクトルートREADMEに書くべきこと

- なぜこの構成なのか（設計原則）
- どうセットアップするか（手順）
- 各ディレクトリの関係性（構成図・依存関係）
- ツール間の参照方式
- トラブルシューティング

#### サブディレクトリREADMEに書くべきこと

- そのディレクトリ内のサブディレクトリ/ファイルの役割一覧
- 新規ファイル追加時の配置判断基準
- 更新手順（そのディレクトリ固有の操作）
- 対応表（外部からの参照マッピング等）

#### 書き分けの原則

1. **ルートREADMEはサブディレクトリの詳細に踏み込まない** — 「詳細は `{dir}/README.md` を参照」で委譲する
2. **サブディレクトリREADMEはルートの情報を繰り返さない** — セットアップ手順やアーキテクチャ全体像はルートに任せる
3. **相互リンクで接続する** — ルートの§9（関連ドキュメント）にサブディレクトリREADMEを列挙する

本ガイドのテンプレート（`~/.shared-ai/templates/readme.md`）はプロジェクトルートREADMEに適用する。サブディレクトリREADMEは上記の原則に従い、必要なセクションのみで構成する。
