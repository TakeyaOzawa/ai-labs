# エージェントプロンプト設計ガイド

scoutエージェントのプロンプト実装時に適用するコンテキスト節約・実行効率化の設計指針。

## プロンプト本体の構造（必須セクション）

以下の順序で記載する。**サイズ目標: 3〜8KB**。

### 共通構造（全エージェント）

```markdown
# {Agent Name}（日本語名）

{1文の概要説明}

## 役割
{対象日/対象の何を、どうやって、何を作るか}
対象領域: {カンマ区切りで列挙}

## スコープ
{担当範囲の明示 + 担当外→他エージェント名}

## 対象日付の決定
{日付決定ロジック}

## 収集手順 / 実行手順
{Phase 0〜N の段階的手順}

## 出力
{ファイルパス + フォーマット（ヘッダのみ簡潔に）}

## 行動原則
{番号付き1行ルール。ドメイン固有の判断基準を含む}
```

### Web検索系エージェント追加セクション

```markdown
## 事前取得済み情報（検索不要）
{RSSフィード取得済みソース一覧}
{RSSでカバーできないサイト一覧}

### Phase 0: 重複排除の準備
{過去3日分レポートからURL抽出 → 既出URLリスト作成}
{トップページ・一覧ページは既出リストに含めない}

### Phase 1: 検索・収集
{一時ファイルパス}
{フィルタリングルール: publishedDate / 既出URL / トップページ}
{検索カテゴリ一覧（番号付き、カテゴリ名のみ）}
{書き出しフォーマット指示}

### Phase 2: レポート生成
{feeds + raw_results 統合 → レポート作成 → 一時ファイル削除}
```

### 評価基準セクション（ドメイン固有）

```markdown
## {関連度基準 / 応用可能性基準}
{⭐⭐⭐ / ⭐⭐ / ⭐ の定義}
```

## コンテキスト節約の設計原則

### プロンプトサイズの制限

| サイズ | 評価 |
|--------|------|
| ~3KB | ✅ 理想的 |
| ~5KB | ✅ 良好 |
| ~8KB | ⚠️ 上限。これ以上は分割を検討 |
| 10KB+ | ❌ コンテキスト枯渇リスク。必ず軽量化する |

### コンパクト化の原則

| 方針 | 具体例 |
|------|--------|
| 検索クエリ例は書かない | カテゴリ名のみ記載。エージェントが適切なクエリを生成する |
| 出力フォーマットはヘッダのみ | 各セクションの見出し + 1行説明。完全なテンプレートは不要 |
| 冗長な説明は箇条書きに圧縮 | 「以下の手順で〜」→ 番号付きリスト |
| 同じ目的の記述は統合 | ただし役割が異なる場合（リスト構築時 vs フィルタ適用時）は分離を維持 |
| ドメイン固有の詳細はreferencesに分離 | ソース一覧が長い場合は `references/{name}-sources.md` へ |
| 行動原則は1行1原則 | 番号付きリストで簡潔に。説明文は不要 |

### 収集対象ソースの記載方法

プロンプト本体には以下の形式で簡潔に記載する:

```markdown
## 事前取得済み情報（検索不要）
`.tmp_{YYYY-MM-DD}_feeds.md` に格納済み。Phase 1ではこれらと重複しないソースに集中すること。
事前取得済み: {ソース1}, {ソース2}, ...
RSSでカバーできないサイト（検索で補完）: {サイト1}, {サイト2}, ...
```

ソース一覧が長い場合（10件超）は `references/{name}-sources.md` に分離し、プロンプトには代表的なもののみ記載する。

## 2段階実行方式（Phase 0 / Phase 1 / Phase 2）

Web検索を伴うエージェントは必ずこの方式を採用する。

### Phase 0: 重複排除の準備

過去3日分のレポートからURLを抽出し「既出URLリスト」を作成する。

除外対象外（既出リストに含めない）:
- パスが `/` で終わるURL（トップページ）
- `/archive/`・`/feed/`・`/news/`・`/blog/`・`/research/`・`/latest` で終わる一覧ページURL

### Phase 1: 検索・収集

```
一時ファイル: .tmp_{YYYY-MM-DD}_raw_results.md
検索回数制限: 最大15回
1回検索するごとに即座にfsAppendで書き出す
検索結果をコンテキストに保持し続けない
```

**フィルタリングルール（各検索結果に適用）:**
- publishedDateが対象日±1日の範囲外 → 除外（null/未提供の場合は通す）
- 既出URLリストに含まれる → 除外
- トップページ（パスが `/` のみ、または上記一覧パターンで終わる）→ 除外

### Phase 2: レポート生成

1. 事前取得フィード `.tmp_{YYYY-MM-DD}_feeds.md` を読み込む（存在すれば）
2. 検索結果 `.tmp_{YYYY-MM-DD}_raw_results.md` を読み込む
3. 統合してレポート作成
4. 完了後、一時ファイルを削除

## 複数テーマ処理時の逐次処理

1つのエージェントが複数テーマを処理する場合:
- **1テーマ完了→ファイル書き出し→次テーマ**の順で逐次処理
- 全テーマの情報を同時にコンテキストに保持しない
- 各テーマの出力は個別ファイルとして書き出す

## テーマ分割実行の判断ポイント

以下のいずれかに該当する場合、**1テーマ=1サブエージェント（別コンテキスト）** に分割する:

| 判断基準 | 閾値 | 例 |
|----------|------|-----|
| テーマ数 × 1テーマあたり検索回数 | 合計30回超 | 5テーマ × 8検索 = 40回 → 分割 |
| 1テーマあたりの出力サイズ | 5KB超/テーマ | 素材シート（裏どり+コードサンプル付き）→ 分割 |
| テーマ選定と収集の性質差 | 選定=軽い / 収集=重い | テーマ選定エージェント + 収集エージェント×N に分離 |
| 実行時に3テーマ目以降で停止した実績 | 1回でも発生 | 即座にテーマ数削減 or 分割を適用 |

### 分割実行の構成パターン

```
{agent-name}（テーマ選定のみ → テーマリストファイル出力）
  ├── {agent-name}-single-1（テーマ1: 収集+出力）
  ├── {agent-name}-single-2（テーマ2: 収集+出力）
  └── ...
       ↓ 全完了後
{downstream-agent}（出力を活用する後続エージェント）
```

### 分割時の影響範囲

| 変更対象 | 内容 |
|----------|------|
| `{agent-name}.md` | テーマ選定のみに縮小。選定結果をファイル出力して終了 |
| 新規: `{agent-name}-single.md` | 1テーマ分の収集に特化したプロンプト |
| `create-{frequency}-tasks.py` | 動的に子タスクを追加するか、固定N件の子タスクを生成 |
| `run-{frequency}-pipeline.py` | `AGENTS` 配列と `NOTIFY_FILE_MAP` に追加 |

### 分割時の懸念と対策

| 懸念 | 対策 |
|------|------|
| テーマ間の重複回避 | 選定フェーズで「互いに異なる技術領域から選ぶ」を強く指示 |
| 推奨順位の最終判断 | 選定フェーズで仮順位を付与。後続エージェントがfrontmatterで最終判断 |
| パイプラインの依存チェーンが深くなる | `run-{frequency}-pipeline.py` の `AGENTS` 順序と `depends_on` で対応可能（追加実装不要） |
| 動的タスク生成の複雑さ | 固定N件（例: 5件）で生成し、テーマが少ない場合はスキップする方が単純 |

## 操作のコマンド抽象化

### 目的

1. **将来のクラウド移行に備える**: 操作をコマンドで抽象化し、移行時の改修をスクリプト内部に限定
2. **コンテキスト逼迫を防ぐ**: コマンド1行で済む形にしてコンテキスト消費を最小化
3. **改修範囲を限定する**: ロジック変更時にプロンプトやhookの書き換えが不要

### 抽象化対象と対応コマンド

| 操作カテゴリ | コマンド | 備考 |
|---|---|---|
| タスク検索 | `python3.12 ~/scripts/find-task.py` | `--pipeline`, `--status`, `--task-name`, `--limit` |
| タスク更新 | `python3.12 ~/scripts/update-task.py` | `--task-file`, `--task-id`, `--scope`, `--set` |
| タスク生成 | `~/scripts/create-{freq}-tasks.sh` | 基準日を引数に受け取り、タスクファイル+RSS事前取得を実行 |
| RSSフィード取得 | `~/scripts/fetch-rss-feeds.py` | `--category`, `--date`, `--output`, `--no-filter` |

### 設計ルール

1. エージェントプロンプト内でファイルパスを直接操作する記述は最小限にする
2. 3回以上同じパターンで使われる操作はスクリプトに切り出す
3. スクリプトの出力はJSON形式を標準とする
4. スクリプトは冪等に設計する
