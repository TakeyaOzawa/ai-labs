# Tasks フォーマットガイド

specのtasks.mdを作成する際は、本ガイドのフォーマットに従ってください。
タスク構造はWBS（Work Breakdown Structure）の考え方に基づき、追加タスクの追跡性を確保します。

## 準拠する考え方

- WBS（Work Breakdown Structure）— タスクの階層分解
- IEEE 29148 Traceability — 要件からタスクへの追跡性
- ISO/IEC 14764:2022 — ソフトウェア保守分類（タスク追加理由の発見フェーズ分類に活用）

## ISO/IEC 14764 保守分類との対応

タスクの追加理由は「いつ発見したか」（発見フェーズ）で分類するが、「なぜ変更するか」はISO/IEC 14764の保守分類で表現できる。
両方の情報が必要な場合は、タスク変更ログの `詳細理由` カラムにISO/IEC 14764の分類を含めることができる。

| ISO/IEC 14764 分類   | タスクでの表れ方                                         |
| -------------------- | -------------------------------------------------------- |
| Corrective（是正）   | fix specのバグ修正タスク、テスト失敗で発見された追加修正 |
| Adaptive（適応）     | 要件変更に伴う追加タスク、外部API変更への対応            |
| Perfective（完全化） | 実装中に発見された改善タスク、レビュー指摘による品質向上 |
| Preventive（予防）   | 設計時に発見された潜在リスクへの対応タスク               |

## front-matter 仕様

requirements.mdと同一のfront-matterを使用する。
フィールド定義はグローバルsteering `requirements-format-guide.md` を参照。

## タスクの階層構造

Kiroのタスク実行エンジンは2階層（親タスク + サブタスク）を認識する。
フェーズとステップはMarkdown見出しとして人間の可読性のために使用する。

```
### フェーズ1: 基盤構築              ← h3見出し（Kiro無視。論理的なグルーピング）
#### ステップ1.1: DB設計             ← h4見出し（Kiro無視。任意。タスクが多い場合に使用）
- [ ] 1. マイグレーション作成        ← レベル1: タスク（Kiro認識・実行）
  - [ ] 1.1 テーブル定義             ← レベル2: サブタスク（Kiro認識・実行）
```

### 階層の使い分け

| 階層       | 記法                       | Kiro認識 | 用途               | 使用基準                             |
| ---------- | -------------------------- | -------- | ------------------ | ------------------------------------ |
| フェーズ   | `### フェーズN:`           | ×        | タスクの大分類     | 常に使用                             |
| ステップ   | `#### ステップN.N:`        | ×        | フェーズ内の中分類 | 1フェーズに7タスク以上ある場合に使用 |
| タスク     | `- [ ] N. タスク名`        | ○        | 実行単位           | 常に使用                             |
| サブタスク | `  - [ ] N.N サブタスク名` | ○        | タスクの細分化     | 1タスクが複数ファイルにまたがる場合  |

- 3階層目のサブタスク（N.N.N）はKiroが認識しないため使用しない
- ステップ層は任意。1フェーズ内のタスクが少ない場合は省略してよい
- 3階層が必要に感じたら、親タスクを分割する

## タスク記述ルール

### 必須項目

| 項目         | 説明                       | 例                                                  |
| ------------ | -------------------------- | --------------------------------------------------- |
| タスク名     | 何をするかを端的に         | `マイグレーションファイルの作成`                    |
| 対象ファイル | 変更・作成するファイルパス | `database/migrations/YYYY_MM_DD_*.php`              |
| 変更内容     | 具体的な変更内容           | `purchase_customersテーブルにsync_flagカラムを追加` |
| 対応要件     | requirements.mdの要件ID    | `REQ-1, REQ-2`                                      |
| 追加理由     | このタスクが存在する理由   | `計画時` / `実装時: {詳細}`                         |

### チェックボックスの状態

| 記法    | 状態       |
| ------- | ---------- |
| `- [ ]` | 未着手     |
| `- [x]` | 完了       |
| `- [-]` | 進行中     |
| `- [~]` | キュー待ち |

### 必須/オプション

| 記法                 | 種別             |
| -------------------- | ---------------- |
| `- [ ] 1. タスク名`  | 必須タスク       |
| `- [ ]* 1. タスク名` | オプションタスク |

### タスクIDの採番規則

| 種別                   | 形式                   | 例             |
| ---------------------- | ---------------------- | -------------- |
| 初期計画タスク         | 連番                   | `1`, `2`, `3`  |
| サブタスク             | 親ID + 枝番            | `1.1`, `1.2`   |
| 追加タスク             | Aプレフィックス + 連番 | `A1`, `A2`     |
| 追加タスクのサブタスク | A親ID + 枝番           | `A1.1`, `A1.2` |

Aプレフィックスにより、初期計画と追加タスクが一目で区別できる。

## 追加タスクの発見フェーズ分類

タスク追加理由は「いつ・何がきっかけで判明したか」で分類する。
ソフトウェア開発ライフサイクルのフェーズに沿った分類で、MECEを担保する。

| 発見フェーズ | 説明                           | タイミング                | 例                                                 |
| ------------ | ------------------------------ | ------------------------- | -------------------------------------------------- |
| 計画時       | 初期タスク分解で計画           | requirements/design作成時 | `計画時`                                           |
| 設計時       | 設計レビューや設計詳細化で判明 | design.md作成・レビュー時 | `設計時: ER図作成中にリレーション不足を発見`       |
| 実装時       | コーディング中に判明           | タスク実装中              | `実装時: フォーム送信時のバリデーション不足を発見` |
| レビュー時   | コードレビューで指摘           | PR作成・レビュー時        | `レビュー時: N+1クエリの指摘`                      |
| テスト時     | テスト実行で判明               | テスト作成・実行時        | `テスト時: 境界値でのエラーを発見`                 |
| 外部要因     | 要件変更、外部依存、環境変化   | 任意のタイミング          | `外部要因: REQ-3の追加に伴う`                      |

### 分類の根拠

この6分類はソフトウェア開発ライフサイクル（SDLC）のフェーズに対応しており、「どのフェーズで発見されたか」という軸で排他的かつ網羅的（MECE）になっている。

```
計画 → 設計 → 実装 → レビュー → テスト
                                    ↑
                              外部要因（任意のタイミングで発生）
```

ISO/IEC 14764のソフトウェア保守分類（Corrective / Adaptive / Perfective / Preventive）は「なぜ変更するか」の軸であり、本分類の「いつ発見したか」の軸とは直交する。両方の情報が必要な場合は、`詳細理由` カラムにISO/IEC 14764の分類を含めることができる。

## 追加タスクの記録方法

### 1. タスク一覧に追加

```markdown
- [ ] A1. バリデーションルールの追加
  - **対象ファイル:** app/Http/Requests/...
  - **変更内容:** 入力値のバリデーション追加
  - **対応要件:** REQ-1
  - **追加理由:** 実装時: フォーム送信時にバリデーションが不足していた
```

### 2. タスク変更ログに記録

```markdown
## タスク変更ログ

| 日付       | 変更種別 | タスクID | 内容                       | 発見フェーズ | 詳細理由                                 | 対応者 |
| ---------- | -------- | -------- | -------------------------- | ------------ | ---------------------------------------- | ------ |
| 2026-04-15 | 追加     | A1       | バリデーションルールの追加 | 実装時       | フォーム送信時にバリデーション不足を発見 | kiro   |
| 2026-04-16 | 変更     | 2        | 対象ファイル変更           | レビュー時   | サービスクラスへの分離を指摘された       | ozawa  |
| 2026-04-16 | 削除     | 3.2      | 不要なテストケース削除     | テスト時     | 親テストケースでカバー済みと判明         | kiro   |
```

## フェーズ分割の基準

| フェーズ  | 内容                               | 例                                       |
| --------- | ---------------------------------- | ---------------------------------------- |
| フェーズ1 | 基盤（DB、モデル、設定）           | マイグレーション、モデル作成、ルート設定 |
| フェーズ2 | 機能実装（コントローラ、サービス） | CRUD実装、ビジネスロジック               |
| フェーズ3 | テスト実装                         | Feature/Unitテスト                       |
| フェーズ4 | 品質チェック（オプション）         | Pint、Larastan                           |

## タスクの粒度

- 1タスク = 1つの論理的な変更単位（1コミットに相当する粒度）
- 複数ファイルにまたがる場合はサブタスクに分割
- テストは機能実装とは別タスクにする
- 3階層が必要に感じたら、親タスクを分割する

## チェックポイントタスク

フェーズの区切りに配置し、実装の整合性を確認するタスク。
必須項目（対象ファイル、変更内容、対応要件、追加理由）は不要。確認内容のみ記載する。

```markdown
- [ ] N. チェックポイント — {確認内容}
  - {確認する観点を箇条書きで記載}
  - 疑問点があればユーザーに確認する
```

チェックポイントタスクは実装タスクの間に挿入し、次のフェーズに進む前の品質ゲートとして機能する。

## 完了時の記録

```markdown
- [x] 1. マイグレーションファイルの作成
  - **対象ファイル:** database/migrations/2026_04_15_000000_add_sync_flag.php
  - **変更内容:** purchase_customersテーブルにsync_flagカラムを追加
  - **対応要件:** REQ-1
  - **追加理由:** 計画時
  - **実装メモ:** nullable booleanとして追加。デフォルト値はnull。
```

## 変更種別ごとの注意事項

### feat の場合

- requirements.mdの影響範囲（§6）でリグレッションリスクが特定されている場合、フェーズ3にリグレッションテストを含める
- 既存機能への影響がない場合でも、共有テーブル・共有サービスを変更する場合は既存テストの実行確認をタスクに含める

### fix の場合

- フェーズ1にバグ再現テストを含める（修正前に失敗することを確認）
- フェーズ2に修正実装
- フェーズ3にリグレッションテスト（requirements.mdの影響範囲で特定したリスク箇所を対象に含める）

### refactor の場合

- フェーズ1に既存テストの確認（変更前に全テストがパスすることを確認）
- フェーズ2にリファクタリング実装
- フェーズ3に既存テストの更新（必要な場合のみ）

### perf の場合

- フェーズ1にベンチマーク計測（改善前の数値を記録）
- フェーズ2に改善実装
- フェーズ3にベンチマーク再計測（改善後の数値を記録）

## 要件トレーサビリティ

すべてのタスクは `対応要件` フィールドでrequirements.mdの要件IDに紐づける。
追加タスクも必ず対応する要件IDを記載する。要件に紐づかないタスク（品質チェック等）はオプションタスクとする。
