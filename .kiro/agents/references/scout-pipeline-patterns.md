# Scout パイプライン設計パターン

scoutエージェントの実装時に適用するコンテキスト節約・実行効率化のパターン集。
エージェントプロンプト（`*-scout*.md`, `*-writer*.md`, `*-updater*.md`）を読み込んだ際に自動ロードされる。

## コンテキスト節約の設計原則

### ⚠️ 最重要: プロンプトサイズの制限

| サイズ | 評価 |
|--------|------|
| ~3KB | ✅ 理想的（tech-trend-scout相当） |
| ~5KB | ✅ 良好 |
| ~8KB | ⚠️ 上限。これ以上は分割を検討 |
| 10KB+ | ❌ コンテキスト枯渇リスク。必ず軽量化する |

### 軽量化テクニック

1. **収集対象ソースは `references/` に分離**: プロンプト本体にはRSS事前取得済みリストと「カバー外サイト」の1行のみ
2. **出力フォーマットは簡略化**: セクション名 + 各セクションに含める項目を1行で列挙。テンプレート全文は不要
3. **検索クエリ例は `references/` に分離**: プロンプト本体には検索順序と各ステップの代表クエリ1〜2個のみ
4. **行動原則は1行1原則**: 番号付きリストで簡潔に。説明文は不要

## 2段階実行方式（Phase 1 / Phase 2）

Web検索を伴うエージェントは必ずこの方式を採用する:

```
Phase 1: 検索・収集
- 一時ファイル: .tmp_{YYYY-MM-DD}_raw_results.md
- 1回検索するごとに即座にfsAppendで書き出す
- 検索結果をコンテキストに保持し続けない
- 検索回数制限: 最大12〜15回

Phase 2: レポート生成
- 事前取得フィード .tmp_feeds.md を読み込む（存在すれば）
- 検索結果 .tmp_raw_results.md を読み込む
- 統合してレポート作成
- 完了後、一時ファイルを削除
```

## 複数テーマ処理時の逐次処理

1つのエージェントが複数テーマを処理する場合:
- **1テーマ完了→ファイル書き出し→次テーマ**の順で逐次処理
- 全テーマの情報を同時にコンテキストに保持しない
- 各テーマの出力は個別ファイルとして書き出す

## テーマ分割実行の判断ポイント

以下のいずれかに該当する場合、**1テーマ=1サブエージェント（別コンテキスト）** に分割する:

| 判断基準 | 閾値 | 例 |
|----------|------|-----|
| テーマ数 × 1テーマあたり検索回数 | 合計30回超 | 5テーマ × 8検索 = 40回 → 分割 |
| 1テーマあたりの出力サイズ | 5KB超/テーマ | 素材シート（裏どり+コードサンプル付き）→ 分割 |
| テーマ選定と収集の性質差 | 選定=軽い / 収集=重い | テーマ選定エージェント + 収集エージェント×N に分離 |
| 実行時に3テーマ目以降で停止した実績 | 1回でも発生 | 即座にテーマ数削減 or 分割を適用 |

### 分割実行の構成パターン

```
{agent-name}（テーマ選定のみ → テーマリストファイル出力）
  ├── {agent-name}-single-1（テーマ1: 収集+出力）
  ├── {agent-name}-single-2（テーマ2: 収集+出力）
  └── ...
       ↓ 全完了後
{downstream-agent}（出力を活用する後続エージェント）
```

### 分割時の影響範囲

| 変更対象 | 内容 |
|----------|------|
| `{agent-name}.md` | テーマ選定のみに縮小。選定結果をファイル出力して終了 |
| 新規: `{agent-name}-single.md` | 1テーマ分の収集に特化したプロンプト |
| `create-{frequency}-tasks.sh` | 動的に子タスクを追加するか、固定N件の子タスクを生成 |
| `pipeline-executor.md` | 対象タスクリストに追加 |
| 後続エージェントのプロンプト | 入力ファイルパスの参照方法を確認（変更不要な場合が多い） |

### 分割時の懸念と対策

| 懸念 | 対策 |
|------|------|
| テーマ間の重複回避 | 選定フェーズで「互いに異なる技術領域から選ぶ」を強く指示 |
| 推奨順位の最終判断 | 選定フェーズで仮順位を付与。後続エージェントがfrontmatterで最終判断 |
| 引き継ぎ情報の分散 | 選定フェーズで仮の引き継ぎファイルを作成。各singleは個別出力のみ |
| パイプラインの依存チェーンが深くなる | 現行のwatcher hookの再発火方式で対応可能（追加実装不要） |
| 動的タスク生成の複雑さ | 固定N件（例: 5件）で生成し、テーマが少ない場合はスキップする方が単純 |

## 操作のコマンド抽象化

### 目的

1. **将来のクラウド移行に備える**: 操作をコマンドで抽象化し、移行時の改修をスクリプト内部に限定
2. **コンテキスト逼迫を防ぐ**: コマンド1行で済む形にしてコンテキスト消費を最小化
3. **改修範囲を限定する**: ロジック変更時にプロンプトやhookの書き換えが不要

### 抽象化対象と対応コマンド

| 操作カテゴリ | コマンド | 実装 | 備考 |
|---|---|---|---|
| タスク検索 | `~/scripts/find-task.sh` | zsh | `--pipeline`, `--status`, `--task-name`, `--limit` |
| タスク更新 | `~/scripts/update-task.sh` | zsh | `--task-file`, `--task-id`, `--scope`, `--set` |
| タスク生成 | `~/scripts/create-{freq}-tasks.sh` | zsh | 基準日を引数に受け取り、タスクファイル+RSS事前取得を実行 |
| RSSフィード取得 | `~/scripts/fetch-rss-feeds.py` | python | `--category`, `--date`, `--output`, `--no-filter` |
| 一時ファイル管理 | エージェント内で `fsWrite`/`fsAppend`/`deleteFile` | - | 将来的にコマンド化を検討 |

### 設計ルール

1. **エージェントプロンプト内でファイルパスを直接操作する記述は最小限にする**
   - ✅ `~/scripts/find-task.sh --pipeline weekly --status starting --limit 1`
   - ❌ `cat ~/Documents/works/agent_histories/scout_weekly/*.json | jq '...'`
2. **3回以上同じパターンで使われる操作はスクリプトに切り出す**
3. **スクリプトの出力はJSON形式を標準とする**
4. **スクリプトは冪等に設計する**

### 将来のクラウド移行時の想定

```
現在:  ~/scripts/find-task.sh → ローカルJSONファイルをjqで検索
将来:  ~/scripts/find-task.sh → DynamoDB/Firestore APIを呼び出し
```

エージェントプロンプトとhookは変更不要。スクリプトの内部実装のみ差し替える。

### 一時ファイル操作について（現時点では未スクリプト化）

現在、一時ファイル（`.tmp_*`）の操作はKiro writeツールで行っている。理由:
- hook発火は「対象外→即終了」で~300Bのコンテキスト消費に留まり実害が小さい
- スクリプト化の実装コストに対してメリットが限定的

スクリプト化を検討すべき状況:
- パイプラインのタスク数増加でhook発火の累積コンテキスト消費が無視できなくなった場合
- クラウド移行で一時ファイルをS3/GCS等に置き換える必要が出た場合

### 新規スクリプト作成時のテンプレート

```bash
#!/bin/zsh
set -eu -o pipefail

# {script-name}: {1行説明}
# 使い方: {script-name}.sh [オプション]
# 出力: JSON形式

# ─── 引数パース ──────────────────────────────────────────
# ─── メイン処理 ──────────────────────────────────────────
# ─── JSON出力 ────────────────────────────────────────────
echo "{\"success\": true, ...}"
```

## RSS事前取得の設計

### fetch-rss-feeds.py へのカテゴリ追加

```python
# scripts/fetch-rss-feeds.py の FEEDS dict に追加
"{category_name}": [
    {"name": "{フィード名}", "url": "{RSS/Atom URL}", "type": "rss" | "atom"},
]
```

- `--category` の choices にも追加
- `output_dirs` に出力先マッピングを追加
- タスク生成スクリプト（`create-{frequency}-tasks.sh`）にRSS取得コマンドを追加

### プロンプトでの参照

```markdown
## 事前取得済み情報（検索不要）
以下のサイトはRSSフィードで事前取得済み。`.tmp_{YYYY-MM-DD}_feeds.md` に格納されている。
事前取得済み: {フィード名1}, {フィード名2}, ...
RSSでカバーできないサイト（検索で補完）: {サイト1}, {サイト2}, ...
```

## references/ ファイルの記載方法

```markdown
# {Agent Name} — 収集対象ソース参照

## {カテゴリ1}
{ソース名}, {ソース名}, ...

## 検索クエリ例
- {パターン1}: `{クエリ例}`
```

- ソース名のリストのみ。URLは不要（エージェントが検索で見つける）
- 検索クエリ例は代表的なもののみ（全パターン網羅は不要）
